### 🔴 Hard findings

**1. NUL byte (`\x00`) in any check/suite string field → HTTP 500, not 422.**
Reproduced twice:
```
POST /api/v1/suites/{id}/checks
{"name": "nulbyte-\u0000-test", "expectation_type": "expect_column_values_to_not_be_null", "config": {"column": "ID"}}
→ HTTP 500 {"error":{"code":"internal_error","message":"Internal server error","detail":{}}}

POST /api/v1/suites
{"name": "qa-verifier-scratch-nulcheck\u0000", "connection_id": "dce36f51-7f23-4ded-83b9-b66f0eba2ebc"}
→ HTTP 500 (same)
```
Root cause confirmed from `dataq-api-1` logs: `ValueError: A string literal cannot contain NUL (0x00) characters.` — Pydantic's `str` validator (`SuiteCreate.name` at `backend/app/api/v1/suites.py:68`, `CheckCreate.name` at `backend/app/api/v1/checks.py:34`) happily accepts a NUL byte, but the Postgres driver rejects it at INSERT time, and that `ValueError` isn't caught anywhere — it falls through to `_unhandled_exception_handler` in `backend/app/core/errors.py`. This is exactly the class of gap CONTRIBUTING rule 4a's adversarial battery targets (hostile/malformed data-ingesting input on a persisted string field, happy-path-only validation). No data corruption resulted (transaction rolled back cleanly — verified via direct DB query, zero half-committed rows), so it's an availability/error-envelope bug, not a data-integrity bug. **Fix shape:** either a Pydantic field validator stripping/rejecting `\x00` on every free-text input field (name/description/config values), or a narrow `except ValueError` around the commit path that maps a NUL-byte DB rejection to a 422. Given the number of string fields across the API (suite name/description, check name, connection name, notification config, etc.), a shared validator/sanitizer is the more scalable fix than patching each endpoint. **Recommend filing via `/gh-issue-from-finding`** — this should block v1.0.0 tagging per the "never a 500 on bad input" rule.

### 🟡 Concerns

1. **No threshold-ordering/sign validation on generic `expectation`-kind checks.** `derive_status` in `backend/app/services/severity.py:107` documents "thresholds are ordered (warn ≤ fail ≤ critical)" as an assumption, but `create_check`/`update_check` in `backend/app/services/check_service.py` never enforce it for plain `kind='expectation'` checks — only the `freshness`/`volume` monitor-kind path gets threshold validation (`validate_monitor_check`). Inverted (warn=90 &gt; fail=50 &gt; critical=10) and negative thresholds both round-trip as 201. Not a crash, and `derive_status`'s `&gt;=` cascade degrades gracefully with inverted values (just produces surprising severity bands, e.g. a negative-metric check would resolve differently than the author intended) rather than erroring — but it's a silent authoring footgun matching the spirit of the "freshness needs a positive threshold" guard that already exists for monitors. Consider extending that same validation to all threshold-bearing checks.
2. **No expectation-type/config-schema validation on generic `expectation`-kind checks at authoring time.** `check_service.py`'s own docstring says this is deliberate ("per-expectation schema validation against live data is the check dry-run path... not CRUD"), and only custom-SQL + monitor kinds get pre-flight validation. A bogus `expectation_type` (`expect_the_impossible_thing`) or a syntactically-valid-but-semantically-empty `config: {}` both persist as 201. This is a documented design tradeoff, not a bug, but it means any client bypassing the frontend's catalog-driven editor (a raw API call, an MCP tool, `suites/import`) can silently create a check that will only ever resolve `error` at run time, discoverable only after a run. Worth a conscious go/no-go decision rather than an oversight — if intentional, no action needed; if not, it's a straightforward addition (validate `expectation_type` against the GX catalog + minimal arg-shape check, mirroring the monitor path).
3. **Webhook auth hostility (item 6) and duplicate-delivery dedup (#456) could not be exercised live** — local dev stack has no ADF/Airflow webhook secret provisioned (503 `webhook_not_configured` on every call). Code review of `backend/app/api/v1/orchestration.py` shows correct `hmac.compare_digest`-based constant-time comparison with explicit non-ASCII-safe handling, but this is not a substitute for a live 401 assertion. If webhook hostility needs to be in the v1.0.0 gate, provision `ADF_WEBHOOK_SECRET_NAME`/`AIRFLOW_WEBHOOK_SECRET_NAME` in a scratch secret store first.
4. **Cross-user authz probes (item 4: view-only 403, unshared-suite 404/403-not-leak, admin-403-for-non-admin) could not be exercised live** — dev-bypass (`backend/app/core/auth.py`) resolves every HTTP request to one fixed, workspace-admin identity regardless of the seeded `analyst@dataq.local` user. This is architecturally correct for local dev (fail-closed real-mode elsewhere), but it means true IDOR/permission-tier testing needs either a real-mode bearer token for the second demo user or a unit-level (not HTTP-level) test — out of scope for this black-box pass. What *was* verified live: random-UUID suite/connection lookups return clean 404s with no data leaked in the body, and no secret ever appears in any read-back payload.

### Verdict

**Block — 1 hard finding (NUL-byte 500) before tagging v1.0.0.**

Everything else in the hostile-authoring, custom-SQL guardrail, flat-file batch-target, run-lifecycle, monitor-config, and dashboard/query-param batteries returned clean, correctly-coded 4xx envelopes — no other 500s, no hangs, no orphaned rows, no secret leakage, no silently-persisted invalid state.
