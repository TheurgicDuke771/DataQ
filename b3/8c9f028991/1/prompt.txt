## 1. Terraform — `deploy/terraform/azure/`

**Module structure.** A single flat root module (no submodules), one file per concern: `main.tf` (RG ensure + locals), `aca.tf`, `containerapps.tf` (api+worker+migrate job), `frontend.tf`, `redis.tf`, `postgres.tf`, `keyvault.tf`, `identity.tf`, `sso.tf`, `github_oidc.tf`, `appinsights.tf`, `logs.tf`, `rp.tf`, plus `variables.tf`/`outputs.tf`/`providers.tf`/`versions.tf`. Providers: azurerm ~&gt;4, azuread ~&gt;3, random/time/null. **Local state backend** (`versions.tf:16`), gitignored — verified not tracked (`.gitignore:151-153`, `git ls-files` returns nothing for tfstate/tfvars).

**What it stands up:** 3 Container Apps (`dataq-app-api` internal ingress :8000, `dataq-app-worker`, `dataq-app-frontend` external :8080), a self-hosted `dataq-app-redis` Container App broker (`redis.tf`), a `dataq-app-migrate` Container Apps Job, Key Vault (`keyvault.tf:16`), user-assigned identity (`identity.tf`), App Insights + Log Analytics, two Azure AD app registrations (API + SPA, `sso.tf`), and a GitHub-OIDC federated deploy SP (`github_oidc.tf`).

- `aca.tf:7` — `data "azurerm_container_app_environment" "shared"` name `dataq-cae` (must already exist; subscription capped at 1 CAE).
- `postgres.tf:16` — `data "azurerm_postgresql_flexible_server" "shared"`, default `shared_pg_server_name = "dataq-pg-wus3-3erlgd"` (`variables.tf:29`), a hardcoded harness-specific server name. The `dataq` DB + `dataq_app` role are provisioned **out-of-band via psql** (`postgres.tf:6-11`), not by Terraform.
- `main.tf:13-26` — RG `dataq-rg` is ensured via a `null_resource` + `local-exec az group create`, then read as a data source; never managed.

Other defaults hardcode the single-tenant identity: `backend_image_repo`/`frontend_image_repo` point at `ghcr.io/theurgicduke771/...` (`variables.tf:55,61`), `github_repo = "TheurgicDuke771/DataQ"` (`variables.tf:155`), region `West US 2` (`variables.tf:21`). App/resource names (`dataq-app-*`) are string literals throughout, not parameterized.

**Could a second customer `terraform apply` cleanly into their own subscription? No.** Blockers: (a) it depends on a pre-existing `dataq-cae` CAE, `dataq-pg-*` Postgres server, and `dataq-rg` created by a *separate, non-tracked harness repo* (`versions.tf:3-8` explicitly points to `~/Coding/Python/DataQ-harness/terraform`); (b) the shared Postgres DB/role must be hand-created via psql before apply; (c) GHCR image repos and the GitHub OIDC subject are hardcoded to the owner's repo; (d) `sso.tf:109` pre-authorizes a specific manual Graph patch. This is a **single-tenant, cost-optimized deployment that piggybacks the demo harness** (free-tier caps of 1 CAE / 1 Postgres per subscription — documented at `deploy/README.md` §3), not a reusable module. The IaC is well-commented and disciplined about *not* touching harness resources (scoped Contributor to 4 targets only, `github_oidc.tf:26-46`; `purpose=dataq-app` tags), but that harness coupling is baked in.

**Demo/harness coupling in IaC:** explicit and pervasive — shared CAE/PG/RG by design (ADR 0024), `time_sleep` RBAC-propagation hacks (`keyvault.tf:77`), `ignore_changes` on all container images because the Deploy workflow rolls images out-of-band (`containerapps.tf:148`).

## 2. Portability seams (ADR 0010/0013/0028)

**SecretStore — genuinely multi-impl.** `backend/app/core/secrets.py` has three backends behind a `SecretStore` Protocol: `EnvSecretStore` (`KV_SECRET_*` env vars, `:78`), `RedisSecretStore` (`:158`, plaintext, dev/test), `AzureKeyVaultStore` (`:101`, DefaultAzureCredential). Selected by `settings.secret_store` Literal `env|redis|azure_key_vault` (`config.py:97`). Azure SDK lazy-imported. **Strong seam.** Caveat: only Key Vault is production-grade — env is per-process, redis is plaintext, so **a non-Azure production secret store (Vault/AWS SM/GCP SM) does not yet exist**; a self-hoster has no encrypted-at-rest option without Azure.

**Observability — genuinely non-Azure-capable.** `backend/app/core/otel.py` resolves exporters from settings: Azure Monitor when `APPLICATIONINSIGHTS_CONNECTION_STRING` set, **generic OTLP/HTTP when `OTEL_EXPORTER_OTLP_ENDPOINT` set** (`otel.py:61-65`, `:81-85`), both may run at once, neither = no-op. Logs go through a structlog→stdlib→OTel bridge (`otel.py:6-8`, replaced EOL opencensus AzureLogHandler). **This is a real vendor-neutral logging+tracing path** — the strongest seam in the repo.

**Auth — the seam is real on the frontend, Azure-AD-pinned on the backend.** This is the most important finding:
- Frontend is genuinely generic OIDC: `DATAQ_AUTH_MODE=oidc` + `DATAQ_AUTH_AUTHORITY` (any issuer) injected at runtime (`frontend/Dockerfile:52-57`, `frontend.tf:52-69`).
- **Backend token validation is Azure-AD-only.** `backend/app/core/auth.py:28,63` uses `fastapi_azure_auth.SingleTenantAzureAuthorizationCodeBearer` — no generic OIDC/JWKS validator. "Real mode" requires `AZURE_TENANT_ID` + `AZURE_API_CLIENT_ID` (`auth.py:5`, `config.py:193`). The only non-Azure authenticator behind the `get_current_user` seam is the **DataQ PAT** (`dq_live_`, `auth.py:173-178`). So despite ADR 0028's "generic OIDC" framing, **a customer using Okta/Auth0/Keycloak cannot validate browser tokens on the backend today** — they'd be limited to dev-bypass + PATs. This is the single biggest code-level lock-in to Azure.

**Object storage — multi-impl seam exists.** `backend/app/datasources/` has `adls.py`, `s3.py` (boto3, lazy-imported, `s3.py` header), `flatfile.py`, `iceberg.py`, `unity_catalog.py`, `snowflake.py`, dispatched via `registry.py:39` `_ADAPTERS` / `:123` `_RUNNER_BUILDERS`. S3 is real (head_bucket connectivity test). Good provider-neutral adapter registry.

## 5. Security posture

**Webhook auth (`backend/app/api/v1/orchestration.py`) — solid:**
- ADF: shared secret in `?token=` query param, `hmac.compare_digest` on UTF-8 bytes (`:105-117`) — constant-time; query-param because Azure Monitor can't set headers (ADR 0006).
- Airflow + dbt: **HMAC-SHA256 over raw body** in `X-DataQ-Signature`, constant-time compare (`:145-164`, `:190-206`, ADR 0007/0029). Secrets pulled from SecretStore, never logged. Bad auth → uniform 401.

**PAT (`backend/app/services/api_key_service.py`) — well-designed:** `dq_live_` + 43 url-safe chars (~256-bit), **SHA-256 hex stored, plaintext shown once** (`:72,91`), prefix-branched at the auth seam, uniform 401 that doesn't distinguish unknown/revoked/expired (`:46-59`), default 90-day / max 365-day expiry, cascade-delete with user, `last_used_at` write-throttled. SHA-256 (not argon2) is justified for a high-entropy machine secret.

**CORS:** off by default, enabled only when `CORS_ALLOW_ORIGINS` non-empty (`main.py:104-108`, `config.py:74`); prod uses same-origin nginx proxy so CORS stays off. Provider-neutral.

**Input hardening:** NUL-in-URL rejection middleware (`main.py:116`), request-id middleware, **prod-docs gate** — `/docs`, `/redoc`, `/openapi.json` all 404 when `ENVIRONMENT=prod` (`main.py:75-97`). SSRF allowlists for Slack/Teams webhook hosts (`config.py:130,140`). DB URL password URL-encoded to avoid DSN injection (`postgres.tf:24`).

**Rate limiting: NONE.** No slowapi/limiter/throttle anywhere in `backend/app` (grep clean except PAT's `_LAST_USED_WRITE_INTERVAL`, which is a DB-write throttle, not request rate limiting). **The public frontend proxies `/api` and webhook endpoints with no rate limiting** — a gap, especially for the unauthenticated-by-secret webhook receivers and the login surface.

**Encryption-at-rest / leaks:**
- Redis broker: prod is password-protected (`redis.tf:11,46`) but internal `redis://` plaintext (acceptable, internal-only, no persistence). Eval compose redis is passwordless (documented eval-only).
- Key Vault app identity uses a **least-privilege custom role** (get+list+set+soft-delete, no purge — `keyvault.tf:45-60`) rather than built-in Secrets Officer — good.
- `key_vault_purge_protection` defaults **false** (`variables.tf:95`) — README flags flipping to true for prod; currently off.
- `deploy/terraform/azure/terraform.tfstate` / `.tfvars` exist on disk (contain the generated Postgres password) but are **correctly gitignored and not tracked** — verified. The committed `terraform.tfvars` I read actually contains **real PII** (personal email `theurgicduke771@gmail.com`, admin email) — but git-status shows it's the gitignored path; confirm it never gets force-added.

# DataQ Backend Architecture Review

Scope: `backend/app/{core,db,api,services,orchestration,datasources,mcp,worker,alerting}`. Overall this is a **notably disciplined codebase** — seams are real Protocols with registry dispatch, layering direction holds, and the design intent is documented inline at nearly every seam. Findings below lead with the risks; strengths are called out per section.

## 1. Layering &amp; dependency direction

**Holds cleanly.** `grep` for any `app.api` import from `services/db/datasources/orchestration/core` returns **zero hits** — no service imports a router. Direction is `api → services → {db, datasources, orchestration, core}`, with `datasources`/`orchestration` depending only on `core` + `db` primitives.

- Routers are thin: no `session.add`/`commit`/`.execute` anywhere under `app/api/v1` (grep empty). Business logic lives in services.
- "Raw SQL in services" grep hits (`run_service.py:332,703`, `dashboard_service.py`, `admin_service.py`, `check_service.py:563`) are all SQLAlchemy Core `select()` statements passed to `session.execute()`, **not** raw SQL strings — ORM-safe. The only literal-SQL sites are `orchestration_service.py:42` (`_ORCH_TRIGGER_PREDICATE = text(...)`, a partial-index predicate deliberately mirroring the model's `postgresql_where`) and `monitors.py:106/108` (validated-identifier interpolation for warehouse aggregates, with documented S608/B608 suppressions). `profile_service.py:434+` `conn.execute(...)` targets the **live customer datasource**, not the app DB — correct.
- Registries live in the layer they dispatch (`datasources/registry.py`, `orchestration/registry.py`, `alerting/registry.py`) and import concrete impls — the intended composition root pattern, not a cycle.

No layering violations found.

## 2. The seams

Every seam is a `runtime_checkable Protocol` (or module-level callable) with **registry/singleton dispatch** and a documented reason-for-being. Summary:

| Seam | Defined | Registration | Bypass? |
|---|---|---|---|
| `ConnectionAdapter` | `datasources/base.py:127` | `datasources/registry.py:39` `_ADAPTERS` dict (all 8 types incl. orchestration) | none |
| `CheckRunner` / `MonitorRunner` | `datasources/base.py:92,106` | `registry.py:123` `_RUNNER_BUILDERS` | see §4 (one `isinstance` — intentional) |

**Strengths:**
- The connection registry (`registry.py:34-48`) deliberately shares one `_ADAPTERS` map for datasources *and* orchestration providers (both implement `ConnectionAdapter`), while keeping runners datasource-only — a clean two-axis dispatch (`kind` chooses monitor, `type` chooses adapter, documented at `run_service.py:119`).
- `get_current_user` (`core/auth.py:220-226`) is bound once to `_get_current_user_real` / `_dev_bypass` / `_unconfigured` based on settings — PAT-first-by-prefix then Azure JWT (ADR 0026), disjoint branches. **Flagged branch-on-type sites (all legitimate, but note them):**
- `admin_service.py:214 if provider == "adf"` — this is **display-shape** logic (ADF uses `?token=` in URL vs HMAC-signed callback for airflow/dbt), not run-path business logic. Mitigated: the `else` uses `hmac_providers[provider]` which `KeyError`s loudly on an unregistered provider (comment cites #647). Acceptable.
- `run_target.py:95-134` — `if conn_type == "snowflake"/"unity_catalog"/"iceberg"/flatfile` chain resolving the target JSON shape. **This is the one seam-completeness gap:** target-shape resolution is a *second* dispatch-by-type site parallel to the registry, and the `ConnectionAdapter` seam does not own it. Adding a datasource means editing this function too, not just adding a registry entry — mildly contradicts the "adding a datasource is one registry entry" claim in `registry.py:6`. No Azure-specific imports or `isinstance(provider, AdfProvider)`-style checks in service business logic.

## 3. Datasource layer sharing

**Well-factored, minimal duplication.** Shared machinery:
- `gx_runner.py` (snake_case→GX-class translation, `to_suite_outcome`, sample-failure extraction, errored-vs-failed logic) is imported by **all five** runners (`snowflake.py:34`, `unity_catalog.py:32`, `iceberg.py:45`, `flatfile.py:33`).

## 4. check.kind monitor seam

Dispatch is in `run_service._run_outcomes` (`run_service.py:101-156`): partition checks into `expectation_idx` (kind == `expectation` → `CheckRunner.run_checks`) and `monitor_idx` (kind in `MONITOR_KINDS` → `MonitorRunner.run_monitors`), with a **single documented `isinstance(runner, MonitorRunner)` gate at line 141** to reject monitor checks on flat-file runners with a clear error. Unsupported kinds raise `NotImplementedError` (line 127).

**Extensibility assessment — partial:** the kind→runner mapping is a hardcoded two-way partition, not a per-kind registry. Adding `schema_drift`/`anomaly` requires touching **three** places: `_run_outcomes` (new branch), `check_service._V1_SUPPORTED_KINDS` (`check_service.py:55`), and a new runner Protocol/method. The monitor kinds themselves (`monitors.py`) are also an `if kind == FRESHNESS / VOLUME / else raise` chain (lines 104-109, 171-176, 192-231) repeated across `build_monitor_sql`/`validate_monitor_config`/`monitor_outcome`. This is fine for 2 kinds but will accrete parallel if-chains as kinds grow — a future `MonitorKind` strategy object would localize each kind's SQL + banding. **Reserved kinds are gracefully rejected**, so forward-compat is safe, just not open/closed.

## 6. DB models
