### 🟡 Concerns

**1. Lock behavior on `results` (the item you specifically flagged) — real and worth a hard look before merge, file `a9b0c1d2e3f4_results_check_fk_cascade.py:41-42`.**

`op.drop_constraint(name, table, type_="foreignkey")` followed immediately by `op.create_foreign_key(..., ondelete="CASCADE")` runs inside **one Alembic transaction** — `backend/alembic/env.py` uses the default `context.begin_transaction()` with no `transaction_per_migration=False`/autocommit block. That matters because:

- `DROP CONSTRAINT` alone requires `ACCESS EXCLUSIVE` on the table, but it's fast (catalog-only, no scan).
- `ADD CONSTRAINT ... FOREIGN KEY` in isolation only requires `SHARE ROW EXCLUSIVE` (blocks writers, not readers) — **but** it also performs a full validation scan of the referencing table by default (no `NOT VALID`), and that scan runs under whatever lock is held.
- Postgres never *downgrades* a lock mid-transaction. Once the `DROP CONSTRAINT` statement has taken `ACCESS EXCLUSIVE` on `results`, that lock is held for the rest of the transaction — through the subsequent `ADD CONSTRAINT`'s full-table validation scan.

Net effect: for the duration it takes Postgres to scan the *entire* `results` table to re-validate the FK, `results` is `ACCESS EXCLUSIVE`-locked — **blocking every read and write**, not just writes. Any in-flight API request reading `/runs/{id}` results, or a worker writing a new `Result` row, will block for that window; if it exceeds a client/API `statement_timeout`, those calls fail. Given ADR 0024's migrate-job-then-roll deploy model, this happens while the *old* code image is still live and serving traffic — the failure mode isn't "crashes on the new shape," it's "stalls/times out live traffic on the largest table for however long the scan takes."

The re-validation is also somewhat wasted work: the prior `NO ACTION` constraint already guaranteed every existing `results.check_id` / `runs.suite_id` value is valid, but Postgres has no "trust me, skip the scan" flag for `ADD CONSTRAINT` — only `NOT VALID` + a separately-run `VALIDATE CONSTRAINT` gets you out from under the exclusive-scan lock.
