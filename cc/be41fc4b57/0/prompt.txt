## Summary: Email+OTP Signin Discussion

**Starting question:** Could email+OTP replace Basic auth for local/non-Azure-AD users, given DataQ already rejected Basic auth (ADR 0026) because it would make DataQ a password system?

**Recommendation:** Yes, worth pursuing — OTP is passwordless (avoids the exact problem Basic auth was rejected for) and it closes a real gap: BYOL customers on non-Azure clouds currently have *no* way to sign in (Azure AD is the only human-auth path today; PATs need an existing user to mint them). Proposed as a third authenticator behind the existing `get_current_user` seam (`backend/app/core/auth.py`), alongside Azure AD JWTs and PATs — scoped as its own small ADR rather than folded into 0026.

**1. Session/token design**
- Reuse the PAT *mechanism* (opaque, prefixed token — e.g. `dq_sess_…` — SHA-256 hash-at-rest, indexed lookup) rather than have DataQ self-issue JWTs, to avoid owning a signing-key/rotation story.
- Don't store it in the `api_keys` table though — different lifecycle/visibility than user-managed PATs. Give it its own `sessions` table.
- Short, fixed expiry (e.g. 12–24h), no refresh-token pair — since re-running OTP is low-friction, expiry just means "log in again."
- Open item this forces: `users.aad_object_id` needs to become nullable with `email` as an alternate key, since OTP users have no AAD identity (a small preview of ADR 0026's deferred "generic principal" work).

**2. Email delivery**
- The existing alert mailer (`backend/app/alerting/email.py`) is a bad direct fit — it's a best-effort, silently-no-op background publisher; OTP is on the auth critical path and needs real errors surfaced, not swallowed.
- Recommendation: reuse the SMTP+STARTTLS *code shape* (provider-neutral, per ADR 0010) but give OTP its own config block (`AUTH_EMAIL_*`, separate from alerting's `EMAIL_*`) so a misconfigured/rate-limited alert mailer can never block sign-in, and vice versa.
- Flagged: OTP "resend code" is a new abuse vector (mail-bombing, account enumeration) — needs its own per-email/per-IP rate limiting regardless of transport.
- Send synchronously (not via Celery) — SMTP submission is normally sub-second and the endpoint is already rate-limited.

**3. Secret storage**
- SMTP password → same pattern as connection credentials and the alert mailer: a config-driven secret *name*, resolved via the existing `SecretStore` protocol at send time. No new mechanism needed.
- OTP code + session token → **not** `SecretStore` (that's for retrievable secrets). These are verifier secrets, same category as PAT hashes (ADR 0026) — hashed, stored in their own DB table(s), never retrievable.
- Called out a deliberate deviation from the PAT precedent: PATs skip a slow KDF because they have ~256 bits of entropy; a 6-digit OTP code has only ~20 bits, so hashing algorithm choice barely matters — the real protection has to be a hard attempt cap + short TTL on the verify endpoint.

**4. Side finding (pre-existing gap, not OTP-specific):** `SecretStore` is only provider-agnostic *in shape* — Azure Key Vault is the only production-grade backend. `EnvSecretStore` and `RedisSecretStore` are both explicitly dev/test-only (plaintext, not persisted). So a non-Azure production deployment currently has nowhere safe to put *any* secret, including the new SMTP password — same gap connection credentials already have. Flagged as a decision point: bundle a real non-Azure backend (e.g. Vault, AWS Secrets Manager) into this effort, or explicitly defer it as a known, separate gap.

**Status:** Nothing implemented yet — all open design discussion, no ADR drafted, no code changed.

```
The model: two axes, one enforcement primitive
There are no stored roles. A user row carries no role column and there's no roles table — authorization is derived at request time from two sources:

Axis 1 — workspace-admin (global, config-driven). WORKSPACE_ADMIN_EMAILS is a comma-separated env allowlist, matched case-insensitively against the IdP-supplied email (auth.py:229-241). It's deliberately a generic identity attribute (no Entra claim read, per ADR 0010/0013). Being on the list grants: the /admin read endpoints via require_workspace_admin (403 otherwise), implicit admin on every suite, and workspace-wide visibility (extended to the MCP tools and schedules in #488/#695). There is no in-app management of this — changing admins means changing the env var and restarting.

Axis 2 — suite-level sharing (the only resource-level RBAC). Defined entirely in suite_authz.py, with a strict capability ladder view(1) < edit(2) < admin(3) < owner(4):

Level	How you get it	What it allows
view	a shares row	read the suite, its checks, its results
edit	a shares row	+ create/update/delete checks, update the suite, trigger runs
admin	being a workspace-admin — implicit on every suite, not grantable (ADR 0027)	+ manage shares, delete the suite
owner	being suite.created_by — immutable, can't be revoked or re-granted	same as admin
The shares table (models.py:452) is unique on (suite_id, user_id), cascade-deleted with either side, and its check constraint only permits view/edit for new grants — grantable suite-admin was dropped by ADR 0027 (legacy pre-#482 admin share rows still resolve until the downgrade migration).

Enforcement
Everything funnels through one primitive: require_permission(session, suite_id, user_id, minimum=…) (suite_authz.py:113). Two properties worth knowing:

Existence hiding: no access at all → 404, indistinguishable from "doesn't exist"; some access but below the required level → 403 with have/need in the detail.
Same gate everywhere: the suites/checks/runs/notifications routers and all 8 MCP tools call the identical service-layer check, so REST and MCP can't drift. List endpoints use the batched effective_permissions (single IN query, no N+1) to stamp each suite with the caller's level, which is what the frontend uses to show/hide per-suite actions — the UI only mirrors, the server decides.
What is not permission-scoped (by design, but know it)
Connections are workspace-global. Any authenticated user can list, create, edit, re-auth, delete, and test any connection — created_by is recorded for display only, with no ownership gate (connection_service.py:199-219). The workspace is the trust boundary; a view-only user on every suite can still delete the Snowflake connection everything runs on.
Credentials are never readable through any tier — secrets go to the SecretStore and only resolve server-side at use time.
PATs inherit the owner wholesale (ADR 0026 phase 1) — a token is exactly as powerful as its user; there are no scoped-down or read-only tokens yet.
Known limitations (all recorded, mostly as gap G-e)
No groups/teams, no custom roles, no per-connection ACLs, admin manageable only via env restart, and no data-access audit trail (G-d/#431 — flagged as the prerequisite for regulated deployments). The roadmap's own verdict on this model is "fine internally, not sellable" for multi-team commercial use — it's the single-tenant, high-trust-workspace design working as intended, with Theme 3 / ADR 0026 phase 2 (service accounts, richer principals) as the evolution path.
```
