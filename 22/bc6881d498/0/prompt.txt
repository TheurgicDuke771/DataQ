## Findings

**1. [HIGH — silent prod no-op] `backend/app/main.py:47-48` — `instrument_fastapi(_app)` inside the lifespan never emits a single HTTP span**
Starlette builds `middleware_stack` lazily on the *first ASGI call — which is the lifespan scope itself* — before the lifespan handler body runs. OTel's `instrument_app` (0.64b0) works by patching `app.build_middleware_stack`; since the stack is already built and cached, the patch is never invoked. **Empirically verified in the repo's conda env: instrumenting inside a lifespan yields 0 spans** (vs. spans when instrumented pre-serve). So in production with `APPLICATIONINSIGHTS_CONNECTION_STRING` set, the API half of A3 ships as a silent no-op — no request spans, no error, nothing in App Insights. The Celery half is unaffected (`worker_process_init` fires before any task). Fix: call `configure_tracing("dataq-api")` + `instrument_fastapi(app)` at module scope in `main.py` right after `app = FastAPI(...)` (any point before the first ASGI call works).

**2. [MEDIUM — test gap that hides #1] `backend/tests/core/test_tracing.py:52-108` — the enabled-path test instruments a toy `FastAPI()` before serving, not the shipped wiring**

**3. [MEDIUM — altitude] `backend/app/worker/celery_app.py:117-127` — Celery is only instrumented on the worker side, so request→task traces never link**
`CeleryInstrumentor` creates the PRODUCER span and injects trace context into task headers via its `before_task_publish` hook — which only exists in processes where `.instrument()` ran. The whole logging chain is keyed on `request_id_var` (`logging.py`, propagated FastAPI→Celery per CLAUDE.md §6 observability), but the OTel spans use their own trace ids and the log pipeline is still opencensus — in App Insights there is no field linking a span to the logs of the same request. ## Findings

**1. FastAPI→Celery trace propagation never happens — task spans are orphaned root traces**
- File: `backend/app/worker/celery_app.py:117-127` (and `backend/app/core/tracing.py:80-87`)
- Summary: `instrument_celery()` runs only in `worker_process_init`, so the **API process** never instruments the publish side; `CeleryInstrumentor`'s `before_task_publish` hook (producer span + traceparent header injection) is never registered where `send_task`/`delay` is called.
- Failure scenario: user triggers `POST /suites/{id}/run`; App Insights shows a `dataq-api` request trace and a separate, unparented `dataq-worker` task trace with no operation-ID linkage. The stated point of A3 (and the CLAUDE.md "request_id correlation FastAPI → Celery → GX" rule, which this feature ports to traces) silently doesn't hold — the custom `request_id` header still flows, but the distributed trace doesn't. Fix: also call `instrument_celery()` in the API startup path (publish-side hooks are safe there; the module already guards on `_provider`).

**2. Unanchored `healthz` in EXCLUDED_URLS drops spans for any URL containing the substring**
- File: `backend/app/core/tracing.py:38`
- Summary: `parse_excluded_urls` applies `re.search` per comma-part against the full URL, so bare `healthz` matches anywhere in host or path — verified: `ExcludeList.url_disabled('http://h/api/v1/suites/healthz-suite/run')` → `True` on the installed opentelemetry-util-http.
- Failure scenario: a suite/connection/check whose slug or name-in-path contains "healthz" (or a deployment hostname containing it — every URL is built as `scheme://host/path`, so a host like `healthz-probe.internal` excludes *all* spans) silently emits no server spans. Anchor it, e.g. `EXCLUDED_URLS = r"/healthz$,/api/v1/orchestration/events/"` (the ASGI exclusion check receives the URL *without* the query string, so `$` after `healthz` is safe).

**3. Non-excluded server spans record the full query string in `http.url` — PII bypasses the logger-level redactor**
- File: `backend/app/core/tracing.py:70-76` (`instrument_fastapi`)
- Summary: verified on 0.64b0 that `collect_request_attributes` emits `http.url = 'http://h/api/v1/users?q=olivia@corp.com'`; OTel's built-in sanitizer only redacts a fixed list (`sig`, `Signature`, `AWSAccessKeyId`, `X-Goog-Signature`), not arbitrary params.
- Failure scenario: the user-search endpoint (`backend/app/api/v1/users.py:45`, `?q=&lt;email or name substring&gt;`) and any future PII-bearing filter param land verbatim in App Insights span attributes — exactly the channel the repo deliberately closed in logs (`request_id_middleware` logs `request.url.path` only, per #494, and PII redaction lives in `logging.py` which spans never traverse). The webhook exclusion covers the *token* case but not the *PII* class. Fix: pass a `server_request_hook` that strips/redacts the query from `http.url`/`http.target`.

**4. Enabled-path test permanently poisons the process-global tracer provider for the rest of the pytest session**
- File: `backend/tests/core/test_tracing.py:79-81` (via `configure_tracing` → `trace.set_tracer_provider`)
- Summary: the test sets the once-only global provider (wrapping a `BatchSpanProcessor` + `InMemorySpanExporter`) and never shuts it down; the autouse fixture resets only `tracing._provider`, not the OTel global.
- Failure scenario: any *later* test in the same pytest process that legitimately needs to install a tracer provider (e.g. a future integration test of the worker path, or a second enabled-path test someone adds in another file) gets a silently-ignored `set_tracer_provider` ("Overriding of current TracerProvider is not allowed" warning) and asserts against the wrong provider; the leaked batch-export thread also keeps running. The docstring acknowledges the constraint within *this* file but nothing protects the rest of the suite. Consider `provider.shutdown()` + module-scoped isolation, or injecting the provider without touching the global in tests.

## The three "also check" questions

- **`CeleryInstrumentor` vs beat / `task_always_eager`:** No fork-safety problem — with `worker -B` (both dev compose line 104 and prod `containerapps.tf:185`), embedded beat is forked from the worker **main** process, which never runs `worker_process_init` and therefore holds no provider/BatchSpanProcessor to inherit (correctly avoiding a #405-style fork bug). Under `task_always_eager` in tests, `worker_process_init` never fires, so tasks run uninstrumented — no bad interaction. The only consequence is Finding 1's flip side: beat-scheduled and eager tasks produce no producer spans, which is acceptable.
- **Idempotence guard:** Safe in practice — `get_settings()` is `lru_cache`d, so the connection string can't differ between calls within a process; and when conn is unset, `_provider` stays `None` so a later call would correctly retry. One benign nuance: if some other code had already called `trace.set_tracer_provider`, the second set is silently ignored, but since both instrumentors receive `tracer_provider=_provider` explicitly, spans still route to the right provider. Not a finding.
- **EXCLUDED_URLS semantics:** the comma-split + `re.search`-on-full-URL model works for the webhook exclusion (verified `.../events/adf?token=s` is excluded, and exclusion happens before span creation so the token never reaches attributes), but the unanchored `healthz` part is Finding 2.
