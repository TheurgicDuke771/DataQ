## Summary: Email+OTP Signin Discussion

**Starting question:** Could email+OTP replace Basic auth for local/non-Azure-AD users, given DataQ already rejected Basic auth (ADR 0026) because it would make DataQ a password system?

**Recommendation:** Yes, worth pursuing — OTP is passwordless (avoids the exact problem Basic auth was rejected for) and it closes a real gap: BYOL customers on non-Azure clouds currently have *no* way to sign in (Azure AD is the only human-auth path today; PATs need an existing user to mint them). Proposed as a third authenticator behind the existing `get_current_user` seam (`backend/app/core/auth.py`), alongside Azure AD JWTs and PATs — scoped as its own small ADR rather than folded into 0026.

**1. Session/token design**
- Reuse the PAT *mechanism* (opaque, prefixed token — e.g. `dq_sess_…` — SHA-256 hash-at-rest, indexed lookup) rather than have DataQ self-issue JWTs, to avoid owning a signing-key/rotation story.
- Don't store it in the `api_keys` table though — different lifecycle/visibility than user-managed PATs. Give it its own `sessions` table.
- Short, fixed expiry (e.g. 12–24h), no refresh-token pair — since re-running OTP is low-friction, expiry just means "log in again."
- Open item this forces: `users.aad_object_id` needs to become nullable with `email` as an alternate key, since OTP users have no AAD identity (a small preview of ADR 0026's deferred "generic principal" work).

**2. Email delivery**
- The existing alert mailer (`backend/app/alerting/email.py`) is a bad direct fit — it's a best-effort, silently-no-op background publisher; OTP is on the auth critical path and needs real errors surfaced, not swallowed.
- Recommendation: reuse the SMTP+STARTTLS *code shape* (provider-neutral, per ADR 0010) but give OTP its own config block (`AUTH_EMAIL_*`, separate from alerting's `EMAIL_*`) so a misconfigured/rate-limited alert mailer can never block sign-in, and vice versa.
- Flagged: OTP "resend code" is a new abuse vector (mail-bombing, account enumeration) — needs its own per-email/per-IP rate limiting regardless of transport.
- Send synchronously (not via Celery) — SMTP submission is normally sub-second and the endpoint is already rate-limited.

**3. Secret storage**
- SMTP password → same pattern as connection credentials and the alert mailer: a config-driven secret *name*, resolved via the existing `SecretStore` protocol at send time. No new mechanism needed.
- OTP code + session token → **not** `SecretStore` (that's for retrievable secrets). These are verifier secrets, same category as PAT hashes (ADR 0026) — hashed, stored in their own DB table(s), never retrievable.
- Called out a deliberate deviation from the PAT precedent: PATs skip a slow KDF because they have ~256 bits of entropy; a 6-digit OTP code has only ~20 bits, so hashing algorithm choice barely matters — the real protection has to be a hard attempt cap + short TTL on the verify endpoint.

**4. Side finding (pre-existing gap, not OTP-specific):** `SecretStore` is only provider-agnostic *in shape* — Azure Key Vault is the only production-grade backend. `EnvSecretStore` and `RedisSecretStore` are both explicitly dev/test-only (plaintext, not persisted). So a non-Azure production deployment currently has nowhere safe to put *any* secret, including the new SMTP password — same gap connection credentials already have. Flagged as a decision point: bundle a real non-Azure backend (e.g. Vault, AWS Secrets Manager) into this effort, or explicitly defer it as a known, separate gap.

**Status:** Nothing implemented yet — all open design discussion, no ADR drafted, no code changed.
