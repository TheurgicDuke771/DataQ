### HIGH — Telemetry setup is not isolated; any OTel error crashes API + every worker on startup

**Location:** `backend/app/core/logging.py:99-162` (`_configure_otel_log_export`), called unguarded at `logging.py:222`; blast radius via `main.py:51` (lifespan), `main.py:219` (module scope), `celery_app.py` `setup_logging` + `worker_process_init`.

**Scenario (state → outcome):** Operator sets `OTEL_EXPORTER_OTLP_ENDPOINT` and a malformed `OTEL_EXPORTER_OTLP_HEADERS` (parsed by the OTLP exporter constructor), or the pinned `opentelemetry-sdk`/`azure-monitor-*-exporter` versions drift and `LoggerProvider`/exporter/`_RedactingOTelLogHandler` construction raises. `_configure_otel_log_export` has no `try/except`, so the exception propagates out of `configure_logging`, which:
- in the API, aborts the ASGI `lifespan` → app never serves;
- in the worker, raises inside the `setup_logging` and `worker_process_init` signal handlers → worker/child dies → crash-loop.

**Hidden errors this reintroduces:** exactly the #405 blast radius (orchestration polling, scheduled dispatch, gap recovery, purge all stop) — this time as a loud crash-loop instead of a silent stall, but still a full outage caused by a *telemetry* misconfiguration. **User impact:** a bad observability env var takes down the product. There is no degradation to stdout-only logging, which is the whole point of an observability sidecar being best-effort.

**Recommendation:** wrap only the bridge attach in a `try/except Exception`, log via the already-attached stdout handler (`logging.getLogger(__name__).error(...)` / a structured `logForDebugging`-equivalent with an error id), and continue. ### MEDIUM — Bridge attached to the root logger → OTel's own export-failure logs feed back into the pipeline

**Location:** `logging.py:162` (`root.addHandler(handler)`); no exclusion of the `opentelemetry`/`azure` logger namespaces anywhere in `configure_logging`.

**Scenario:** the OTLP collector or App Insights is unreachable (the common failure). The SDK's `BatchLogRecordProcessor`/exporter emits `logger.warning("Failed to export …")` on the `opentelemetry` logger. That record propagates to root → the OTel `LoggingHandler` on root → gets queued for export → the next export also fails → another "failed to export" log → … A self-referential amplification that kicks in *precisely when the backend is down*. It's bounded by the batch interval/queue-drop, so not a tight infinite loop, but it's a log/CPU amplifier and pollutes the very backend you're trying to reach once it recovers.

**Recommendation:** set `logging.getLogger("opentelemetry").propagate = False` (and consider `azure.monitor`) in `configure_logging`, or attach the bridge selectively rather than to root. This is a well-known OTel root-bridge footgun.

# Silent-Failure &amp; Error-Handling Audit — PR #639 (`feature/per-suite-slack-email`)

I verified every claim against the actual diff and the surrounding code, and ran two probes (the light email validator, and Python's `EmailMessage` header handling) to calibrate impact. Overall the secret-lifecycle and SSRF work is solid and the isolation is well-behaved — but there is one genuine silent-failure vector and a couple of state-persistence footguns.

## Finding 1 — HIGH: Light email validation accepts recipients that permanently, silently break email delivery for a suite

**Location:** `backend/app/services/notification_service.py` `assert_valid_recipients` (~lines 110–125); interacts with `backend/app/alerting/email.py:167-169` and `backend/app/alerting/composite.py:27-37`.

**Issue:** The validator only checks each comma-part for one `@`, a non-empty local part, and a `.` in the domain. I confirmed by execution that it **accepts** all of these:

- `a@b.com\nBcc: evil@x.com` and `a@b.com\r\nBcc: evil@x.com` (embedded CR/LF)
- `a@ b.com` (embedded space)
- `&lt;script&gt;@b.com`, `a@b.com;c@d.com`

The security-injection angle is *contained* — Python's `EmailMessage.__setitem__` raises `ValueError("Header values may not contain linefeed or carriage return characters")` when `message["To"]` is set (I verified this), so Bcc header injection does **not** actually reach the wire. But that is precisely what makes this a silent failure:

**Concrete scenario (input/state → wrong outcome):**
1. A user with `edit` on a suite saves `email_recipients = "ops@corp.com\nfoo"` (a paste artifact / typo). Validation **passes** → API returns 200 → the UI shows **"Notifications saved"** and the "Email recipients" tag/state reflect success.
2. On the *next* suite run, `EmailPublisher.publish` reaches `message["To"] = ", ".join(recipients)` → `ValueError`.
3. `CompositePublisher` swallows it under `except Exception` and logs `channel_publish_failed` (composite.py:31).
4. **Net effect:** email alerting for that suite is now permanently dead — every run silently fails to notify — with the only signal being a server log line the suite owner never sees. The user is convinced alerting is configured.

The `a@ b.com` / `&lt;script&gt;@b.com` variants are worse in a subtler way: they don't raise at header-set, so `send_message` will attempt delivery to a malformed/garbage address, producing an SMTP error (again isolated/swallowed) or misrouting.

**Hidden errors:** `ValueError` from the email header machinery; `smtplib.SMTPRecipientsRefused` / `SMTPDataError` from malformed addresses — all funneled into the single broad `channel_publish_failed` bucket and indistinguishable from a transient mailer outage.

**User impact:** False "saved" confirmation, then a silent, ongoing notification outage. This is the exact "validation gives false confidence, downstream failure is swallowed" anti-pattern.

**Recommendation:** Tighten `assert_valid_recipients` to reject control characters and internal whitespace before storing, and prefer a structural parse over the hand-rolled `partition('@')`:

```python
import re
from email.headerregistry import Address

_CTL = re.compile(r"[\r\n\t]")

def assert_valid_recipients(raw: str) -&gt; None:
    parts = parse_recipients(raw)
    if not parts:
        raise InvalidRecipientsError("at least one recipient is required")
    for addr in parts:
        if _CTL.search(addr) or " " in addr:
            raise InvalidRecipientsError(
                "recipient contains invalid characters", detail={"invalid": addr}
            )
        local, sep, domain = addr.partition("@")
        if not sep or not local or "." not in domain:
            raise InvalidRecipientsError(
                "each recipient must be a valid email address", detail={"invalid": addr}
            )
```

Separately, the `String(1024)` column has no length guard in validation — a very long paste will 500 at commit rather than 422; cap it in the validator.

## Finding 2 — MEDIUM: "Clear Teams" / "Clear Slack" silently commit unsaved `enabled` / `alert_on` edits

**Location:** `frontend/src/components/suites/NotificationsPanel.tsx` — the shared `put()` helper and the clear buttons (`onClick={() =&gt; put({ webhook: '' }, ...)}`).

**Issue:** `put()` always sends `{ enabled, alert_on: alertOn, ...extra }` from current in-memory form state. The clear buttons therefore piggyback whatever the user has toggled but **not** yet saved.

**Concrete scenario:** User flips the **Enabled** switch off (or changes the threshold from `warn` to `always`), does **not** click Save, then clicks **"Clear Teams"** because they changed their mind about the webhook. The PUT persists `enabled: false` (or the new threshold) as a side effect of clearing the webhook. The toast says only "Teams webhook cleared." This pattern pre-dates #639 (the old single `onClearWebhook` did the same), but #639 multiplies the surface to two clear buttons and adds a third editable field, so it's worth fixing now.

**Recommendation:** Have the clear buttons send only the channel field plus the *server-known* enabled/alert_on (from `initial*`, not the live edited state), or disable the clear buttons while the form is dirty and prompt the user to Save/Discard first.

## Finding 3 — LOW/MEDIUM: Send-time SSRF re-check drops the `https` scheme check on the workspace webhook path

**Location:** `backend/app/alerting/slack.py:133-136`.

**Issue:** The send-time re-check validates only the host (`host == a or host.endswith(f".{a}")`), not the scheme. Per-suite Slack webhooks are scheme-validated at write time (`_host_allowed` requires `https`), but the **workspace** `SLACK_WEBHOOK_SECRET_NAME` is never write-validated at all. A workspace webhook stored as `http://hooks.slack.com/...` would pass the send-time host check and be POSTed over cleartext.

**Impact:** Admin-controlled config only, so low exploitability — but it's an asymmetry with the write path and defeats the "defense in depth at the sink" intent. (The Teams publisher very likely has the same shape; worth checking for parity.)

**Recommendation:** Re-check scheme too at the sink — reuse `notification_service._host_allowed(webhook, self._allowed_hosts)` (which already enforces `https`) instead of the inline host-only comparison, so both per-suite and workspace webhooks get the identical guard.

## Finding 4 — LOW: Vault secret orphaned on a failed/partial multi-channel `set`; `SecretWriteError` surfaces as an opaque 500

**Location:** `backend/app/services/notification_service.py` `_apply_secret_webhook` (called twice in `upsert_config`, ~lines 220–233); `backend/app/core/secrets.py:55` (`SecretWriteError(Exception)`).

**Issue:** `secret_store.set()` runs **before** `session.commit()`. If the Slack `set` raises `SecretWriteError` after the Teams `set` already succeeded (or the later commit fails), the Teams secret is left written in the vault with no committed DB row referencing it — an orphan. And because `SecretWriteError` is a plain `Exception` (not a `DataQError`), it hits the generic handler and returns an opaque **500** rather than an actionable envelope.

**Note:** This is the documented write-through pattern and pre-dates #639 for the single Teams webhook; #639 doubles the exposure. Importantly, the **clear** path is correct — refs are captured and soft-deleted only *after* commit, so no *live* secret is orphaned on clear (see the positives below).

**Recommendation:** Consider wrapping `SecretWriteError` into a `DataQError` (e.g. 502/503) so a vault write blip is a clean, retryable error instead of a 500; and note the set-before-commit orphan as a known residual (a periodic reconcile of `suite-notif*` refs vs. rows would close it).

## Finding 5 — LOW: Validation `detail` (which address / allowed hosts) is not surfaced to the user

**Location:** `frontend/src/api/client.ts:32-34` maps only `error.message`, not `error.detail`.

**Issue:** `assert_valid_recipients` returns `detail={"invalid": addr}` and `assert_allowed_slack_webhook` returns `detail={"allowed_hosts": [...]}`, but the toast shows only the generic message ("each recipient must be a valid email address"). ### Finding 1 — MEDIUM: stale-target race — a policy derived from the OLD target is silently persisted onto a suite whose target has since changed
`backend/app/worker/tasks.py`, `_auto_classify_columns` (the `target = suite.target` capture through `set_column_policy`).

The task captures `target = suite.target`, introspects that target (seconds of live warehouse I/O), then before persisting only re-checks `suite.column_policy is not None` — it never re-checks that `suite.target` is still the target it profiled. `set_column_policy` writes only `column_policy`, not `target`.

Scenario (inputs/state → wrong outcome): Suite created targeting `SALES.ORDERS` (policy-less) → task dispatched, begins introspecting ORDERS. While it profiles, the owner PATCHes the target to `SALES.CUSTOMERS` (update path: `column_policy is None` still true → a *second* task dispatched too, but that's benign). Task A finishes, `session.refresh(suite)` shows `column_policy is None`, and it writes an `identifier_column`/`pii_columns` derived from ORDERS onto a suite now pointing at CUSTOMERS. Result: the redaction policy references columns that may not exist in the current target — `identifier_column` silently resolves to no locator, and any ORDERS-specific PII columns are the wrong set for CUSTOMERS. The PII value-signal floor at redaction time still masks affirmatively-sensitive values, so this is a wrong-locator / mis-scoped-policy bug, not a PII leak. The guard should re-verify `suite.target == target` (or dict-equal) after `refresh`, not just the policy.

### Finding 2 — LOW/MEDIUM: the "never clobber a user policy" re-check is TOCTOU, not race-safe
`backend/app/worker/tasks.py` — `session.refresh(suite)` + `if suite.column_policy is not None` immediately before `set_column_policy`.

The comment says this makes the write race-safe. It does not. `set_column_policy` issues a plain ORM `UPDATE suite SET column_policy = …` with no `WHERE column_policy IS NULL` and no `SELECT … FOR UPDATE` row lock. Under READ COMMITTED there remains a window between the `refresh` SELECT and the `set_column_policy` COMMIT. A user who sets their own policy via the Auto-detect/set endpoint in that sub-second window has it silently overwritten by the auto-derived one. The re-check narrows the window from "seconds of introspection" to "milliseconds between refresh and commit" — a real and worthwhile mitigation — but the code and its comment overstate it as fully safe. A conditional update (`UPDATE … WHERE column_policy IS NULL` and check rowcount) or a `with_for_update` fetch would close it. Low probability, but the outcome is a user's explicit choice being clobbered by background work — exactly the class of silent override this guard exists to prevent.

### Finding 3 — LOW: target change on an already-policied suite leaves a stale policy with no signal
`backend/app/api/v1/suites.py`, `update_suite` dispatch condition `payload.target is not None and suite.target is not None and suite.column_policy is None`.

Because of the `column_policy is None` clause, PATCHing the target of a suite that already has a policy does not re-derive and emits no warning/event. Scenario: suite targets ORDERS with a derived `identifier_column="ORDER_NUMBER"`; owner repoints it to a table without that column. The stale policy persists silently; the locator just stops working. This is defensible as the intended "never re-derive once a policy exists" rule, but the staleness is invisible to the user. Worth a follow-up issue (or at least a doc note) rather than a code change in this PR.

### Finding 4 — LOW (broad-except caveat, not a defect): a genuine code bug is masked as a best-effort warning
`backend/app/worker/tasks.py`, the task's `except Exception`.

Correct for I/O/introspection failures (warehouse unreachable, bad creds, `ProfileUnsupportedError`/`ProfileTargetInvalidError` 422s) — those *should* be swallowed here. But the same catch also absorbs any programming bug in `suggest_policy_for_target` / `set_column_policy` (e.g. a future `AttributeError`/`TypeError`), reporting it as an ordinary `auto_classify_failed` warning. The feature would then silently never classify anything, distinguishable only by log volume. It is logged at `warning` with `exc_info`, not surfaced as an error/Sentry-level event, so a persistent regression could run unnoticed. Acceptable for best-effort background work, but this is the debugging cost of the broad catch — I'd note it, not block on it.
