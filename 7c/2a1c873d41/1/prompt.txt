### Theme 3 — Asset-first IA (the "asset" concept), design-captured 2026-07-04

## 5. Explicit scope decisions / anti-goals

1. **Do not build a proprietary lineage graph engine.** `context/post-v1-roadmap.md:524-525`: pull lineage from governance catalogs / OpenLineage "**instead of building a lineage graph ourselves**." This directly supersedes the earlier `DataQ_platform_roadmap.md:393` sketch ("visual dependency graph … React Flow … ~1 week"), which is no longer the plan.
2. **Assets are a separate concept from suites, not a replacement.** Suites keep owning execution (scheduling, triggers, notifications, authz); assets own browsing/health/lineage/incidents/blast-radius (`context/post-v1-roadmap.md:207-215`). Explicitly rejected: "making assets primary for *everything*" — cited reasons: suite-scoped authz (ADR 0027) re-keying cost, scheduling/triggers being batch-level concepts, and GX's suite-shaped execution grain.
3. **Asset identity is the OpenLineage naming spec, not an internal ID scheme** — chosen specifically to make future OpenLineage emission/pull "automatic instead of a mapping layer" (`:223-227`).
4. **dbt orchestration seam deliberately does not couple to any host API** (Snowflake tasks, Databricks Jobs, dbt Cloud API) — ADR 0029 alternatives-considered section (`docs/adr/0029:61-65`): "dbt under the Airflow provider — rejected," "Bind to a host API — rejected: each couples observation to one vendor, violating ADR 0010/0013." This keeps the artifact-based (and eventually lineage-capable) surface vendor-neutral.
5. **Sequencing anti-goal: incidents/lineage is explicitly not being built before its prerequisite.** "One migration serves all four [lineage, asset view, incidents, catalog sync] — schedule it before any of them" — i.e., no lineage attach work should start before the asset entity phase 1 lands (`:220-221`).
6. **#596 itself is scoped as a design doc only**, producing next-cycle issues — not an implementation task in v1.1. Confirmed by both the roadmap doc and progress.md exit-gate wording ("design doc … merged with next-cycle phase-1 issues filed").
7. No ADR yet exists dedicated to lineage/assets (checked `docs/adr/README.md` in full — only ADR 0029 dbt-provider and ADR 0030 Iceberg touch adjacent ground; ADR 0030 only tangentially connects by noting native Iceberg snapshot/schema history is "the natural feed for the reserved `freshness`/`schema_drift` monitor kinds," not lineage). The asset/lineage ADR is expected to originate from #596's output.

## 2. `app/orchestration/` — dbt provider (ADR 0029, #609/#611)

`app/orchestration/dbt.py` (319 lines). Confirmed via full read + repo-wide grep for `manifest.json`/`depends_on`:

- **Only reads `run_results.json`** (`_RUN_RESULTS_RELPATH = "latest/run_results.json"`, dbt.py:37, dbt.py:103). **`manifest.json` is never referenced anywhere in the codebase** (grep across `app/orchestration/` and `app/services/orchestration_service.py` returned zero hits).

- `list_recent_runs()` (dbt.py:278-319) builds a `RunUpdate` per job from `doc["metadata"]` (`invocation_id`, `generated_at`, `invocation_started_at`) and `doc.get("results", [])` — no `depends_on`, no model/test node names, no `nodes`/`manifest` keys are extracted.

**Conclusion:** the dbt provider is status/timing-only. Adding lineage would require a new artifact reader for `manifest.json` (node ids, `depends_on.nodes`, resource_type, database/schema/alias per node) — no scaffolding for this exists yet, but the `_read_artifact()` scheme-dispatch helper (dbt.py:102-173, handles file/adls/s3) could be reused/generalized to also fetch `manifest.json` from the same `artifacts_uri`.

## 5. Authz summaries (ADR 0027 + ADR 0033)

<result>## PART A — dbt `manifest.json` as a lineage source

### 1. Where the dependency graph lives

| Key | What it holds |
|---|---|
| `nodes` | Dict of all **models, tests, seeds, snapshots, analyses, operations** (`on-run-start`/`on-run-end` hooks land here too), keyed by `unique_id`. Each entry carries `resource_type`, `depends_on.nodes` (+ `depends_on.macros`), `database`, `schema`, `alias`, `name`, `relation_name`, `config.materialized`, `package_name`, `path`/`original_file_path`, `checksum`, `columns`, `description`. |
| `sources` | Dict of source-table defs, same identity shape (`database`, `schema`, `name`, `relation_name`) but no `depends_on` (sources are graph roots). |
| `parent_map` | `unique_id → [first-order parent unique_ids]` for every node (models, tests, sources-as-keys don't appear as keys since they have no parents, but they appear as *values*). |
| `child_map` | Inverse of `parent_map`: `unique_id → [first-order children]`. This is the fast way to walk a source or model **forward**. |
| `disabled` | Nodes marked `enabled: false` — excluded from `nodes`/`sources` but listed here; usually irrelevant for lineage. |
| `metadata.dbt_schema_version` | A URL, e.g. `https://schemas.getdbt.com/dbt/manifest/v12.json` — self-describing version pointer. |

`parent_map`/`child_map` are the cheapest way to build a graph without re-deriving it from every node's `depends_on.nodes`, since they're already flattened and include sources.

**`unique_id` format:** `&lt;resource_type&gt;.&lt;package_name&gt;.&lt;name&gt;` for models/sources (source ids are `source.&lt;package&gt;.&lt;source_name&gt;.&lt;table_name&gt;`, one dot-segment deeper), tests append a short hash suffix, e.g. `test.dataq_lineage.not_null_mart_order_revenue_order_number.7a14e08b69`. This is confirmed against our own project's manifest (below).

**Physical identity** — `database` + `schema` + `alias` (the actual object name after any `alias()`/config override) together give the physical relation; `relation_name` is dbt's own precomputed `database.schema.alias` string (quoted per adapter), so in practice **`relation_name` alone is the table-level lineage node key** and you don't need to reconstruct it from the three parts yourself — except for ephemeral models, where `relation_name` is `null` (see §3).

Source: [Manifest JSON — dbt Developer Hub](https://docs.getdbt.com/reference/artifacts/manifest-json), [dbt JSON Schemas](https://schemas.getdbt.com/).

Docs pages were vague/contradictory on the exact version↔release mapping, so I pip-downloaded the actual `dbt-core` wheels for 1.5.0 → 1.10.0 and grepped the `@schema_version("manifest", N)` decorator on the `WritableManifest` class (source of truth — this literally sets what gets written):

| dbt-core version | manifest schema version |
|---|---|
| 1.5.x | v9 |
| 1.6.x | v10 |
| 1.7.x | v11 |

- `metadata.dbt_schema_version`, `metadata.dbt_version` (for logging/diagnostics)

- `parent_map` / `child_map` (skip re-deriving from `depends_on` — these are provided pre-flattened)
- Do **not** depend on `compiled_code`, `raw_code`, `columns`, `description`, `docs`, `meta` — these are large, optional for lineage, and the parts most likely to shrink/move (see the size discussion below and the open dbt-core discussion to split descriptions out of manifest.json).

### 3. Sources vs models vs tests — table-level edges, and collapsing ephemeral models

- **Sources** are graph *roots* — they never appear as a `key` in `parent_map` (no parents), only as values in other nodes' parent lists, and as keys in `child_map`.
- **Models** appear in `nodes` with `resource_type: "model"`; their `depends_on.nodes` lists immediate parents (sources and/or other models).
- **Tests** (`resource_type: "test"`) are graph *leaves* in a lineage sense — they depend on one or more models/sources but nothing depends on them. For table-level lineage they should be filtered out entirely (they don't produce a table); they're only useful if you also want "which model has data-test coverage" for DQ purposes.
- **`operation`** resource type appears for `on-run-start`/`on-run-end` hooks (our own project has 6 of these from the `grant ...` hooks in `dbt_project.yml`) — also filter out, not lineage-relevant.

**Deriving table-level edges:** walk `parent_map` per model/source node, filtering `nodes` to `resource_type in {model, seed, snapshot}` ∪ all of `sources`, and emit `(parent.relation_name, node.relation_name)`.

**Ephemeral models** — verified in dbt-core source (`dbt/contracts/graph/nodes.py`):

```python
    return self.config.materialized == "ephemeral"
```
An ephemeral model gets **no physical relation** — it's inlined as a CTE into whatever references it, and its `relation_name` is `null`. To collapse it out of a table-level edge list: when building edges, if a parent (or the node itself) `is_ephemeral`, don't emit an edge for it — instead recurse through *its* `depends_on.nodes` (via `parent_map`) until you reach a non-ephemeral model or a source, and connect directly to that. This is exactly the same recursive-substitution dbt itself does when it inlines ephemeral CTEs at compile time.

(Not an issue in our local test project — all 6 models are `view` or `dynamic_table`, none `ephemeral`.)

### 4. Size / streaming concerns

- Our own 6-model / 4-source / 5-test project produces a **611 KB** `manifest.json` (dominated by `compiled_code`/`raw_code` per node, not the graph metadata).
- Real-world large projects: a GitHub issue/discussion thread reports **~3,500 models → ~75 MB** manifests, with the dbt team acknowledging the file "necessarily scales linearly with objects + lines of code," and an open discussion (dbt-core #11080) proposing splitting `description` text out of the manifest because it dominates size in well-documented large projects. Astronomer/Cosmos has filed perf issues where parsing a large manifest per-worker adds 50s+ cold-start latency.
- **Implication for DataQ:** a lineage reader should (a) stream-parse (`ijson` or similar) rather than `json.load()` the whole file when project size is unknown/large, and (b) only pull the minimal-subset fields from §2 — never hold `compiled_code`/`raw_code`/`docs` in memory for the whole graph. dbt's own roadmap direction (Fusion / dbt "2.0") is moving toward Parquet artifacts for exactly this reason — worth a forward-looking note in the design doc, but not needed for the current v12 JSON path.

Sources: [Slim down manifest.json — dbt-core #2169](https://github.com/fishtown-analytics/dbt/issues/2169), [Split descriptions from manifest.json — dbt-core discussion #11080](https://github.com/dbt-labs/dbt-core/discussions/11080).

# Lineage integration options for DataQ — research report

**Premise:** DataQ pulls lineage from external systems (for blast-radius questions like "which downstream tables are affected by this failing check?") and emits OpenLineage events (so external catalogs can display DataQ's quality results), rather than building its own lineage graph. Constraints: MIT-licensed project (no GPL/AGPL/SSPL/BUSL/Elastic/source-available deps per ADR 0031), vendor-neutral, local-first (docker-compose reference stack), small team.

## 1. OpenLineage emission from DataQ (`openlineage-python`)

**License — Apache-2.0, clean.** The OpenLineage repo LICENSE is Apache-2.0 ([github.com/OpenLineage/OpenLineage/blob/main/LICENSE](https://github.com/OpenLineage/OpenLineage/blob/main/LICENSE)) and PyPI declares `Apache-2.0` for `openlineage-python`, latest **1.51.0 released 2026-07-06** — actively maintained ([pypi.org/project/openlineage-python](https://pypi.org/project/openlineage-python/)). The project is an LF AI &amp; Data **Graduate** project (since Sept 2023) ([lfaidata.foundation blog](https://lfaidata.foundation/blog/2023/09/20/lf-ai-data-foundation-announces-graduation-of-openlineage-project/)). Fully compatible with DataQ's MIT policy.

**Transports.** HTTP (retry + compression), Console (the default when unconfigured), File (local or S3/GCS/Azure via the `[fsspec]` extra), Kafka (`[kafka]` extra), plus Composite, Transform, Async-HTTP, and custom transports. Config via constructor, `openlineage.yml`, or env vars — legacy `OPENLINEAGE_URL`/`OPENLINEAGE_API_KEY` or the namespaced `OPENLINEAGE__TRANSPORT__TYPE=http` style; `OPENLINEAGE_DISABLED` is the global kill switch ([openlineage.io/docs/client/python/configuration](https://openlineage.io/docs/client/python/configuration)). The Console default + `OPENLINEAGE_DISABLED` means DataQ can ship the emitter dark, off-by-default, with zero infra dependency.

**Event model.** `RunEvent` = one observation of a `Run` (UUID `runId`) of a `Job` (namespace + name) with `inputs`/`outputs` `Dataset` lists; `eventType` ∈ START/RUNNING/COMPLETE/ABORT/FAIL/OTHER — the spec expects exactly one START and one terminal event per run ([openlineage.io/docs/spec/object-model](https://openlineage.io/docs/spec/object-model), schema: [spec/OpenLineage.json](https://raw.githubusercontent.com/OpenLineage/OpenLineage/main/spec/OpenLineage.json)).

**Effort to emit "DQ suite run X read dataset Y" — low, verified code shape** ([openlineage.io/docs/client/python/usage](https://openlineage.io/docs/client/python/usage)):

```python
from openlineage.client import OpenLineageClient
from openlineage.client.event_v2 import Dataset, Job, Run, RunEvent, RunState
from openlineage.client.uuid import generate_new_uuid

client = OpenLineageClient()  # config from env/openlineage.yml
job = Job(namespace="dataq", name="suite.retail_orders")
run = Run(runId=str(generate_new_uuid()))
client.emit(RunEvent(eventType=RunState.START, eventTime=..., run=run, job=job, producer="dataq"))
client.emit(RunEvent(eventType=RunState.COMPLETE, eventTime=..., run=run, job=job, producer="dataq",
                     inputs=[Dataset(namespace="snowflake://acct", name="RETAIL.ORDERS")]))
```

This maps 1:1 onto DataQ's existing run lifecycle (`runs` table start/complete/error) — the emit hooks belong in `run_service`. The only design work is the **dataset naming convention** (OpenLineage namespace/name per datasource type, e.g. `snowflake://&lt;account&gt;` + `db.schema.table`), which the spec standardizes per platform.

**Quality facet — purpose-built for DataQ's use case.** `DataQualityAssertionsDatasetFacet` (facet key `dataQualityAssertions`, schema [openlineage.io/spec/facets/1-1-0/DataQualityAssertionsDatasetFacet.json](https://openlineage.io/spec/facets/1-1-0/DataQualityAssertionsDatasetFacet.json)) attaches an `assertions[]` array to an input dataset: required `assertion` (e.g. `not_null`, `row_count`) + `success` (bool); optional `column`, `name`, `severity` (`error`|`warn` — maps directly to DataQ's severity tiers), `description`, `expected`, `actual` ([facet docs](https://openlineage.io/docs/spec/facets/dataset-facets/data_quality_assertions)). dbt tests and the Great Expectations integration (`OpenLineageValidationAction` on a GX checkpoint) already populate it ([GX integration docs](https://openlineage.io/docs/integrations/great-expectations/)) — so DataQ emitting it from its own GX results is idiomatic, not novel. The older `DataQualityMetricsInputDatasetFacet` (rowCount, per-column nullCount/distinctCount/min/max) is a natural carrier for DataQ's `metric_value` monitor-kinds (freshness/volume).

## Recommended first slice (small team, local-first, MIT-safe)

### (b) Quality-facet publishing — build this FIRST

1. **Emit OpenLineage RunEvents with `DataQualityAssertionsDatasetFacet` from `run_service`** using `openlineage-python` (Apache-2.0, one pip dep, zero infra). ### (a) Blast-radius lineage pull — build SECOND, behind a seam

5. **DataHub impl of the same seam — deferred:** best-in-class external-assertion push API and a native OL HTTP receiver (so slice 1 already works against it), Apache-2.0 SDK; but the Kafka+OpenSearch+MySQL footprint (8 GB RAM minimum) disqualifies it from the compose reference stack. ## 1–2. Per-datasource namespace + name construction, with case/quoting/defaults notes

| Datasource | Namespace template | Name template | Concrete example |
|---|---|---|---|
| **Snowflake** | `snowflake://{organization_name}-{account_name}` (preferred, orgname-account format) **or** `snowflake://{account_locator}(.{region}).{cloud}` (legacy locator format) | `{database}.{schema}.{table}` | `snowflake://myorg-myaccount` / `MYDB.PUBLIC.ORDERS` — legacy form: `snowflake://xy12345.us-east-2.aws` / `MYDB.PUBLIC.ORDERS` |
| **Databricks Unity Catalog** | `unitycatalog://{host}` or `unitycatalog://{host}:{port}` | `{catalog}.{schema}.{table}` | `unitycatalog://adb-123456789.0.azuredatabricks.net` / `main.sales.orders` |
| **ADLS Gen2 (abfss://)** | `abfss://{container_name}@{service_name}.dfs.core.windows.net` | `{path}` | `abfss://raw@mydatalake.dfs.core.windows.net` / `retail/orders/2026/07/10/` |
| **AWS S3** | `s3://{bucket_name}` | `{object_key}` | `s3://dataq-lake` / `retail/purchase_orders/part-00000.parquet` |
| **Apache Iceberg — generic REST catalog** | the catalog's raw REST URI, taken verbatim from `catalog.uri` config (no `iceberg://` prefix; whatever scheme+host the REST endpoint has) | `{iceberg_namespace}.{table}` (Iceberg's own multi-level namespace, dot-joined, exactly as Spark's `Identifier.toString()` renders it) | `https://iceberg-rest.example.com:8181` / `retail.purchase_orders` |
| **Apache Iceberg — Snowflake-hosted Iceberg REST catalog** (special-cased) | `snowflake://{account_identifier}` — reuses the Snowflake namespace rule above, not a generic REST namespace | `{warehouse}.{table}` where `warehouse` = the Iceberg `CatalogProperties.WAREHOUSE_LOCATION` conf, and the whole name is **upper-cased** | `snowflake://myorg-myaccount` / `RETAIL_WH.PURCHASE_ORDERS` |
| **dbt on Snowflake (openlineage-dbt / dbt-ol)** | `snowflake://{fix_account_name(profiles.yml "account")}` — identical normalization function to the spec's account-identifier rule | `{database}.{schema}.{table_or_alias}` (dot-join of non-empty parts, using dbt manifest's own casing) | `snowflake://myorg-myaccount` / `ANALYTICS.MARTS.FCT_ORDERS` |

### Case sensitivity, quoting, and default-schema omission

- **No spec-level case-folding rule exists.** The name components are simply whatever the source system reports (e.g., dbt's `manifest.json` `database`/`schema`/`alias` fields, taken as-is). Practically, this means Snowflake-backed datasets will render **upper-case** unless the underlying objects were created with quoted mixed/lower-case identifiers, because Snowflake itself upper-cases unquoted identifiers — the OpenLineage layer does not re-case anything.
- **Quoting is stripped, not preserved.** `SnowflakeUtils.stripQuotes()` explicitly removes surrounding double-quotes from identifiers before they go into a dataset name/namespace, with the documented rationale: *"double quotes are used as delimiters to preserve case sensitivity or allow special characters in identifiers... quotes are not part of the actual identifier name and should be removed for normalized dataset and column names."* (https://github.com/OpenLineage/OpenLineage/blob/main/client/java/src/main/java/io/openlineage/client/utils/SnowflakeUtils.java)
- **Schema/database omission is engine-driven, not a generic spec default.** The spec doesn't have a universal "default schema" rule — instead, each per-engine name template is fixed at either 2 or 3 segments matching that engine's actual object hierarchy (MySQL/Hive/Teradata have no schema concept, so their template is literally 2-segment). Where a value legitimately can be missing (Snowflake account locator's region/cloud), the account-normalization function inserts an explicit default rather than omitting the segment: bare locator `xy12345` → `xy12345.us-west-1.aws` (defaults to AWS US-West Oregon); `xy12345.us-east-1` → `xy12345.us-east-1.aws` (defaults cloud to `aws`). Source: docstrings in `SnowflakeUtils.parseAccountIdentifier` / `fix_account_name` (both links above).

## 3. Gotchas — cross-environment ambiguity

**Yes — different Snowflake accounts (DEV vs QA) are, by design, distinct namespaces, hence distinct dataset identities.** This isn't just an inference; it's explicit in the spec text itself:

&gt; "Snowflake is a special case where we don't have a single standardized namespace, due to its account identifier model... If you switch formats later, existing lineage nodes won't connect to new ones."
&gt; — https://github.com/OpenLineage/OpenLineage/blob/main/website/docs/spec/naming.md

The maintainer's design rationale (from a live GitHub discussion questioning exactly this kind of ambiguity) draws the line explicitly on **physical/tenant isolation**:

&gt; BigQuery gets a single global namespace (`bigquery`, no project ID) because *"you can query other projects from your project: `SELECT ... FROM project.dataset.table`"* — projects are tenants inside one addressable system. AWS Glue and Spanner *do* fold the account/project ID into the namespace because they're *"physically separate systems"* with regional/instance-level isolation.
&gt; — https://github.com/OpenLineage/OpenLineage/discussions/4251 (maintainer `mobuchowski`)

A DEV Snowflake account and a QA Snowflake account are exactly this second case — physically separate authentication/compute realms reachable at different hostnames — so under the spec's own logic they are *correctly* distinct namespaces (`snowflake://dev-org-account` vs `snowflake://qa-org-account`), not a modeling bug.

**What practitioners do to group "the same logical asset" across environments — the spec deliberately does not solve this at the identifier level:**
- **`SymlinksDatasetFacet`** — the one first-party mechanism for "this is also known as X" — but its documented use case is engine-level aliasing (Hive table addressed by file path vs metastore name; the same dataset seen via FQDN vs short hostname), *not* cross-environment/cross-tenant grouping. Schema: `identifiers: [{namespace, name, type}]`. Source: https://github.com/OpenLineage/OpenLineage/blob/main/spec/facets/SymlinksDatasetFacet.json (rendered: https://openlineage.io/docs/spec/facets/dataset-facets/symlinks/)
- **Custom/consumer-side facets or tags** — the discussion thread's maintainer endorses solving analogous ambiguity (MySQL across multiple hosts/datacenters) via either (a) folding a discriminator into the host-naming strategy itself, or (b) a facet-based extension, calling both "valid" — i.e., there's no canonical spec answer, it's pushed to the implementer.
- **Downstream lineage/catalog tools do the grouping**, not OpenLineage identifiers themselves — e.g., matching same `{schema}.{table}` name across differing namespaces under one business-glossary/logical-asset entity, or tagging datasets with an `environment` custom facet/tag that a UI groups on. This is consistent with `TagsDatasetFacet` existing as a generic escape hatch (https://github.com/OpenLineage/OpenLineage/blob/main/spec/facets/TagsDatasetFacet.json) but there is no spec-blessed "environment" or "logical-asset-id" facet as of this research.

**Design-doc implication for DataQ:** if DataQ wants "flag the same suite/check across DEV/QA/UAT Snowflake" to read as one logical asset in any OpenLineage-facing surface, that grouping needs to live in DataQ's own domain model (e.g., a `logical_asset` tag/facet DataQ attaches), not in the OpenLineage namespace/name — the namespace will correctly and unavoidably differ per account.

## 4. File-pattern datasets (directories, daily-partitioned CSVs, globs)

The spec itself only goes as far as: for every blob/file-store backend (S3, GCS, ABFSS, HDFS, local, remote filesystem, WASBS), **name = `{path}`**, full stop — there is no spec-level notion of a glob or partition wildcard in the name template. Confirmed by re-checking every storage row in the canonical table (https://github.com/OpenLineage/OpenLineage/blob/main/website/docs/spec/naming.md) and by grepping the full facets registry (`spec/facets/*.json`, https://github.com/OpenLineage/OpenLineage/tree/main/spec/facets) — there is **no `PartitionDatasetFacet`** in the first-party facet set.

The convention that fills this gap comes from how the reference Spark integration actually resolves a dataset identifier for a partitioned/pattern read: it uses the relation's/table's base **`location`** (the directory root Spark/Hive resolved the partitioned table or file-source read against), not per-partition-file paths — I confirmed this by reading `PathUtils.java` (https://github.com/OpenLineage/OpenLineage/blob/main/integration/spark/shared/src/main/java/io/openlineage/spark/agent/util/PathUtils.java), which contains no partition- or glob-splitting logic at all; the base directory is what gets passed in as `path` for the `S3`/`HDFS`/`GCS`/`ABFSS` naming classes shown above. So the practitioner/implementation convention is exactly what you hypothesized:

- **Dataset = the directory/prefix** (e.g. `s3://dataq-lake` / `retail/purchase_orders/` — not `retail/purchase_orders/dt=2026-07-10/part-00000.csv`).
- **Partition structure is not modeled as separate datasets.** The two facets available to describe *how* that directory is physically laid out are:
  - `StorageDatasetFacet` — `storageLayer` (`iceberg`/`delta`) + `fileFormat` (`parquet`/`orc`/`avro`/`json`/`csv`/`text`/`xml`) — https://github.com/OpenLineage/OpenLineage/blob/main/spec/facets/StorageDatasetFacet.json
  - `SchemaDatasetFacet` — partition *columns* (e.g. a Hive-style `dt` column) show up here as ordinary schema fields, not as a separate partition concept.
- There is no first-party facet for "list of partition values covered by this run" — teams that need that level of granularity either encode it in a custom facet or in `OutputStatisticsOutputDatasetFacet`/row-count-style facets, outside the core naming spec.

## 5. dbt ↔ OpenLineage (openlineage-dbt / dbt-ol) — exact naming behavior for Snowflake, verified against source

I read the actual method bodies in `integration/common/src/openlineage/common/provider/dbt/processor.py` (the shared processor both `dbt-ol` and the Airflow dbt-Cloud/Core operators build on) rather than relying on docs, since the docs page (https://openlineage.io/docs/integrations/dbt/) doesn't spell this out.

**Namespace** — `DbtArtifactProcessor.extract_namespace(profile)`:
```python
if self.adapter_type == Adapter.SNOWFLAKE:
    return f"snowflake://{fix_account_name(profile['account'])}"
```
`profile['account']` is read straight out of the user's `profiles.yml` target block (dbt's own `account:` key), and `fix_account_name` (https://github.com/OpenLineage/OpenLineage/blob/main/integration/common/src/openlineage/common/provider/snowflake.py) is the **identical** normalization routine used by the core Snowflake naming spec: if the account string contains a hyphen it's treated as already being `orgname-accountname` and passed through unchanged; otherwise it's treated as a legacy locator and gets `.us-west-1.aws` appended (bare locator) or `.aws` appended (locator+region only). This is exactly the same defaulting behavior documented on the canonical Snowflake namespace row.

**Name** — `DbtArtifactProcessor._format_dataset_name(database, schema, table)`:
```python
def _format_dataset_name(self, database, schema, table):
    return ".".join(list(filter(None, [database, schema, table])))
```
called with `node.metadata_node["database"]`, `["schema"]`, and (for models) `["alias"]` or (for sources) `["name"]` — i.e. taken verbatim, in whatever case, from dbt's own `manifest.json`. No re-casing, no quote-stripping happens on the dbt side (unlike the Java `SnowflakeUtils.stripQuotes` used elsewhere in the client).

**Practical implication for matching byte-for-byte:** for DataQ's Snowflake connector to emit dataset identifiers that line up with what `dbt-ol` would emit for the *same* physical tables, it must (a) run the identical account-string normalization (hyphenated org-account passthrough vs. legacy-locator defaulting to `us-west-1`/`aws`), and (b) use the database/schema/table names exactly as Snowflake's catalog returns them unquoted (which will be upper-case for the overwhelming majority of real warehouses) rather than lower-casing or otherwise normalizing them.

Sources: https://github.com/OpenLineage/OpenLineage/blob/main/integration/common/src/openlineage/common/provider/dbt/processor.py · https://github.com/OpenLineage/OpenLineage/blob/main/integration/common/src/openlineage/common/provider/snowflake.py · https://openlineage.io/docs/integrations/dbt/ (env-var config surface, e.g. `OPENLINEAGE_NAMESPACE` for job namespace, `OPENLINEAGE_DBT_USE_EXTENDED_JOB_NAME` for job *name* — neither affects dataset namespace/name, which is always profile/manifest-derived per the above).
