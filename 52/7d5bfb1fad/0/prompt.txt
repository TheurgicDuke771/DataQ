### HIGH — Telemetry setup is not isolated; any OTel error crashes API + every worker on startup

**Location:** `backend/app/core/logging.py:99-162` (`_configure_otel_log_export`), called unguarded at `logging.py:222`; blast radius via `main.py:51` (lifespan), `main.py:219` (module scope), `celery_app.py` `setup_logging` + `worker_process_init`.

**Scenario (state → outcome):** Operator sets `OTEL_EXPORTER_OTLP_ENDPOINT` and a malformed `OTEL_EXPORTER_OTLP_HEADERS` (parsed by the OTLP exporter constructor), or the pinned `opentelemetry-sdk`/`azure-monitor-*-exporter` versions drift and `LoggerProvider`/exporter/`_RedactingOTelLogHandler` construction raises. `_configure_otel_log_export` has no `try/except`, so the exception propagates out of `configure_logging`, which:
- in the API, aborts the ASGI `lifespan` → app never serves;
- in the worker, raises inside the `setup_logging` and `worker_process_init` signal handlers → worker/child dies → crash-loop.

**Hidden errors this reintroduces:** exactly the #405 blast radius (orchestration polling, scheduled dispatch, gap recovery, purge all stop) — this time as a loud crash-loop instead of a silent stall, but still a full outage caused by a *telemetry* misconfiguration. **User impact:** a bad observability env var takes down the product. There is no degradation to stdout-only logging, which is the whole point of an observability sidecar being best-effort.

**Recommendation:** wrap only the bridge attach in a `try/except Exception`, log via the already-attached stdout handler (`logging.getLogger(__name__).error(...)` / a structured `logForDebugging`-equivalent with an error id), and continue. ### MEDIUM — Bridge attached to the root logger → OTel's own export-failure logs feed back into the pipeline

**Location:** `logging.py:162` (`root.addHandler(handler)`); no exclusion of the `opentelemetry`/`azure` logger namespaces anywhere in `configure_logging`.

**Scenario:** the OTLP collector or App Insights is unreachable (the common failure). The SDK's `BatchLogRecordProcessor`/exporter emits `logger.warning("Failed to export …")` on the `opentelemetry` logger. That record propagates to root → the OTel `LoggingHandler` on root → gets queued for export → the next export also fails → another "failed to export" log → … A self-referential amplification that kicks in *precisely when the backend is down*. It's bounded by the batch interval/queue-drop, so not a tight infinite loop, but it's a log/CPU amplifier and pollutes the very backend you're trying to reach once it recovers.

**Recommendation:** set `logging.getLogger("opentelemetry").propagate = False` (and consider `azure.monitor`) in `configure_logging`, or attach the bridge selectively rather than to root. This is a well-known OTel root-bridge footgun.
