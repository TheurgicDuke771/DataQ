```python
def fix_account_name(name: str) -&gt; str:
    """
    Normalize Snowflake account name according to OpenLineage specification.

    Snowflake supports two formats:
    1. Organization-account format (preferred): orgname-accountname
       - Never includes region or cloud in the identifier
       - Example: myorg-myaccount

    2. Account locator format (legacy): accountlocator[.region[.cloud]]
       - Examples:
         - xy12345 → xy12345.us-west-1.aws (AWS US West Oregon default)
         - xy12345.us-east-1 → xy12345.us-east-1.aws (aws default)
         - xy12345.us-east-2.aws → xy12345.us-east-2.aws
         - xy12345.east-us-2.azure → xy12345.east-us-2.azure
    """
    # Split by dots to analyze the structure
    parts = name.split(".")

    # First part is the account identifier
    account_part = parts[0]

    # Check if it's organization-account format (contains hyphen)
    # Organization-account format never has region/cloud info
    if "-" in account_part:
        # Organization-account format - return as-is
        return account_part

    # Account locator format - need to include region and cloud
    if len(parts) == 1:
        # Just account locator, add default region and cloud (AWS US West Oregon)
        return f"{account_part}.us-west-1.aws"
    elif len(parts) == 2:
        # account_locator.region, add default cloud (aws)
        return f"{account_part}.{parts[1]}.aws"
    else:
        # Full format: account_locator.region.cloud
        return name
```

## 1. Run lifecycle — hook points

`/Users/arijit/Coding/Python/DataQ/backend/app/services/run_service.py`

There are **two** entry points that drive a run to `running`, and **three** places a run reaches terminal:

- `execute_run` (line 172-251) — the normal path. START hook: `run.status = "running"` at line 189, committed at line 191 (before the try block). Terminal hooks:
  - success: lines 217-227 (`run.status = "succeeded"`, `finished_at`, `failure_reason = None`, `session.commit()`)
  - cooperative-cancel-during-execution: lines 217-220 and 230-234 (returns early, run stays `cancelled` — **not** a COMPLETE/FAIL case per the issue's own scope, which only lists START/COMPLETE/FAIL)
  - failure: lines 228-243 (`except Exception`, `run.status = "failed"`, `failure_reason = classify_failure_reason(exc)`, `session.commit()`, then `return run` at line 243)
- `skip_run` (line 254-276) — the "no data to check" path. START hook: line 264-266 (`run.status = "running"`, commit). Terminal (always succeeds): lines 272-274 (`run.status = "succeeded"`, `finished_at`, commit).

There is **no single terminal choke point inside `run_service`** — `execute_run`'s success/failure branches and `skip_run`'s success are three separate commit sites. The closest thing to a single choke point is one level up, in the worker:

`/Users/arijit/Coding/Python/DataQ/backend/app/worker/tasks.py`
```python
@celery_app.task(name="run_suite")
def run_suite(run_id: str) -&gt; str:
    rid = uuid.UUID(run_id)
    session = get_session()
    try:
        outcome = _run_suite(session, run_id=rid)
        alert_dispatch.publish_run_outcome(session, run_id=rid)   # &lt;-- lines 193-194
        return outcome
    finally:
        session.close()
```
`alert_dispatch.publish_run_outcome` (below) is exactly this kind of terminal choke point already, and is the closest existing precedent for "one place, after the run reaches a terminal state, fire a best-effort side-effect." But it only fires after `_run_suite` returns — i.e. after **all** of `_run_suite`'s paths (setup-failed, materialize-failed, skip, or `execute_run`) have already committed their terminal status. It does **not** cover the START event (that has to be hooked inside `execute_run`/`skip_run` directly, or by wrapping `run.status = "running"` assignment sites).

`_run_suite` (tasks.py lines 85-176) also has two more terminal-failed sites before `execute_run` is even reached — `_terminal_failed` (lines 66-82, used at line 129 for setup failure and line 152 for materialize failure). These never call `execute_run`, so if OL START must correspond 1:1 with an actual run attempt, these early failures need their own consideration (arguably no START was ever emitted for them, so no FAIL either, or a START+FAIL pair right there — spec decision needed).

Suite/checks access in `_run_suite`: `suite = session.get(Suite, run.suite_id)` (tasks.py:115), `checks = list(session.scalars(select(Check).where(Check.suite_id == suite.id)))` (tasks.py:120). Post-#757, `run.asset_id` is already stamped at construction time by `run_dispatch.new_queued_run`:
```python
# app/services/run_dispatch.py:41-46
return Run(
    suite_id=suite.id,
    asset_id=suite.asset_id,
    status="queued",
    triggered_by=triggered_by,
)
```
So inside `execute_run`/`skip_run`, `run.asset_id` is available directly off the `Run` object passed in — no extra query needed for the id itself, but the **Asset row** (namespace/name) is a separate table and does need a fetch (see §3).

## 2. Results shape at completion — what's in memory, no re-query needed

`_build_result` (`run_service.py:57-95`) builds one `Result` per check from a `CheckOutcome`, all in the `rows` list built at `run_service.py:209-212`:
```python
rows = [
    _build_result(run.id, check, check_outcome)
    for check, check_outcome in zip(checks, outcomes, strict=True)
]
session.add_all(rows)
```
This `rows: list[Result]` (unpersisted-but-populated ORM objects) is exactly what's in memory at the moment the run completes — the emitter can build its facets straight from this list plus the parallel `checks: list[Check]` (already in scope), with **zero extra DB query**. Result fields (`app/db/models.py:485-513`): `status` (str, one of `RESULT_STATUSES`), `metric_value` (Decimal|None), `observed_value`/`expected_value`/`sample_failures` (JSONB dict|None). `Check` fields (`models.py:347-380`): `name`, `kind` (`expectation`/`freshness`/`volume`/...), `expectation_type`, `warn_threshold`/`fail_threshold`/`critical_threshold`, `config` (JSONB).

Severity vocabulary (`app/db/models.py:37-71`):
```python
RUN_STATUSES = ("queued", "running", "succeeded", "failed", "cancelled")
_RESULT_SEVERITY_TIERS = ("pass", "warn", "fail", "critical")
_RESULT_OPERATIONAL_STATUSES = ("skip", "error")
RESULT_STATUSES = _RESULT_SEVERITY_TIERS + _RESULT_OPERATIONAL_STATUSES
SEVERITY_RANK: dict[str, int] = {
    tier: rank for rank, tier in enumerate((t for t in _RESULT_SEVERITY_TIERS if t != "pass"), 1)
}
FAILING_TIERS: tuple[str, ...] = tuple(SEVERITY_RANK)   # ("warn","fail","critical")

def worst_severity(statuses: Iterable[str]) -&gt; str | None:
    present = [s for s in statuses if s in FAILING_TIERS]
    return max(present, key=lambda s: SEVERITY_RANK[s]) if present else None
```
`worst_severity` is already imported into `run_service.py` (line 43) and used at `run_service.py:345` for the runs-list rollup — the emitter's per-run/run-level `severity` (if a single run-level severity is wanted) should reuse this exact helper rather than reimplementing an ordering.

`resolve_status`/`extract_metric` (`app/services/severity.py:33-123`) is the single place that derives `(status, metric_value)` — already called at `run_service.py:73-78` inside `_build_result`. The metrics facet (`metric_value` monitor kinds — freshness/volume) reads `Result.metric_value` directly; `check.kind in MONITOR_KINDS` (`app/datasources/monitors.py:33-35`: `FRESHNESS = "freshness"`, `VOLUME = "volume"`, `MONITOR_KINDS = (FRESHNESS, VOLUME)`) is the discriminator for which checks get the metrics facet vs. the plain assertions facet.

`skip_run` (`run_service.py:267-270`) builds a different, simpler shape — every check gets `status="skip"`, `observed_value={"reason": reason}` — the emitter needs to handle this case too (no assertion outcome really applies; `skip` isn't in `FAILING_TIERS` so `worst_severity` naturally ignores it).

`namespace`+`name` is exactly the OL Dataset identity (see `Asset` docstring, `models.py:162-173`, and `asset_identity.py`'s `resolve_asset_identity`/`AssetIdentity(namespace, name)`).

Loading it in the worker session: `Run.asset_id` is a nullable FK to `assets.id` (SET NULL on delete, `models.py:467-469`). In `_run_suite` (tasks.py), nothing currently loads the `Asset` row — only `run.asset_id` (a bare UUID) exists on the `Run`. The emitter needs `session.get(Asset, run.asset_id)` (same idiom as `session.get(Suite, run.suite_id)` at tasks.py:115 / `session.get(Run, run_id)` at tasks.py:102) guarded for `None` (targetless suite / unresolved asset → `run.asset_id is None` → no input Dataset, or skip emission of that facet — fail-soft per the `Asset` FK comment "SET NULL so an asset sweep never deletes a run", `models.py:466`).

Note `FakeSession` in the test suite (`tests/services/test_run_service.py:20-49`) has **no `.get()` method** — only `add_all`/`commit`/`rollback`/`refresh`. If the emitter hook lives inside `execute_run`/`skip_run` and needs to fetch the `Asset` row itself, either (a) `FakeSession` gains a `.get`, or (b) — cleaner — the caller (`_run_suite` in tasks.py, which already has a real session) resolves the `Asset` once and passes it into `execute_run`/`skip_run`/the emitter as a parameter, keeping `run_service`'s pure/fake-testable contract intact. This is a design decision the spec needs to make explicit.

## 5. Settings conventions &amp; the OTel dark-by-default gating precedent

`/Users/arijit/Coding/Python/DataQ/backend/app/core/config.py:8-19`:
```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env.app",
        env_file_encoding="utf-8",
        extra="forbid",
        case_sensitive=False,
    )
```
`extra="forbid"` is documented (comment, lines 10-14) as catching **typo'd/stale keys in `.env.app`** — it does not reject arbitrary process-environment variables that don't correspond to a declared field. I verified this empirically:
```
$ OPENLINEAGE__TRANSPORT__TYPE=http TOTALLY_UNRELATED_ENV_VAR_XYZ=1 python3 -c \
    "from backend.app.core.config import Settings; print(Settings(_env_file=None).environment)"
dev   # no error
```
So a standard, non-`DATAQ`-prefixed env var like `OPENLINEAGE__TRANSPORT__TYPE` or `OPENLINEAGE_DISABLED` will **not** trip `extra=forbid` as long as it isn't also written into `.env.app` under a name that happens to collide with (but not match the type of) a Settings field. It simply won't be read by `Settings` at all — which is fine, because `openlineage-python`'s own `OpenLineageClient()`/`OpenLineageConfig` reads those vars itself; DataQ's `Settings` only needs its own DataQ-prefixed toggle for the "is emission configured at all" gate (e.g. an `openlineage_transport_url: str | None = None` or reuse of `OPENLINEAGE_URL`/`OPENLINEAGE_DISABLED` presence-checked via `os.environ` directly, mirroring how `secrets.py`'s `EnvSecretStore` reads `KV_SECRET_*` straight from `os.environ` rather than through `Settings`).

**The OTel exporter seam is the closest precedent for the dark-by-default gate itself** (`app/core/otel.py` docstring, lines 1-23, and `build_span_exporters`/`build_log_exporters`, lines 51-79):
```python
"""...
Two exporters sit behind the seam and **may both be active at once**:
- **Azure Monitor** — when ``APPLICATIONINSIGHTS_CONNECTION_STRING`` is set.
- **Generic OTLP/HTTP** — when ``OTEL_EXPORTER_OTLP_ENDPOINT`` is set (#589) ...

Neither set ⇒ telemetry is a complete no-op (the pre-#524 ``AzureLogHandler`` gate).

All exporter imports are **lazy** (repo convention, see ``secrets.py``) so a
deployment that uses neither backend never pays the import cost.
"""
...
def build_span_exporters(settings: Settings | None = None) -&gt; list[SpanExporter]:
    settings = settings or get_settings()
    exporters: list[SpanExporter] = []
    conn = settings.applicationinsights_connection_string
    if conn:
        from azure.monitor.opentelemetry.exporter import AzureMonitorTraceExporter
        exporters.append(AzureMonitorTraceExporter(connection_string=conn))
    base = _otlp_base(settings)
    if base:
        from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
        exporters.append(OTLPSpanExporter(endpoint=f"{base}/v1/traces"))
    return exporters
```
And the empty-list-means-off short-circuit in the caller (`app/core/logging.py:105-108`):
```python
settings = get_settings()
exporters = otel.build_log_exporters(settings=settings)
if not exporters:
    return
```
This "config value present → lazily import + construct → append to a list; caller returns/no-ops on empty list" is the pattern the OL emitter should mirror: `_build_openlineage_client(settings) -&gt; OpenLineageClient | None`, returning `None` when unconfigured, with the emit call sites doing `if client is None: return` before touching anything OL-related.

## 6. PII discipline — what must never leave the app

The redaction seam lives in two places that must both stay untouched by the emitter's design: (1) `core/logging.py`'s structlog `_redact_pii` processor (`app/core/logging.py:14-46, 66-77`), which redacts **log lines** by key-name (`password`, `token`, `email`, etc.) and by regex over string values (secret query-params, URL userinfo credentials) — this governs anything the emitter logs on a fail-open catch, not the OL payload itself. (2) `run_service.redact_sample_failures` (`app/services/run_service.py:587-656`) and its docstring (lines 594-615), which governs what may cross the **read API** boundary for `Result.sample_failures` specifically.

Critically, **`sample_failures` is the only Result column that can carry real row data** — the `Result` model comment says it plainly (`app/db/models.py` — see also `run_service.py:679-681`: "`sample_failures` is the only result column that can carry real (possibly PII-bearing) data rows"). By contrast, `observed_value`/`expected_value` are documented and, by inspection of `_build_result` (`run_service.py:57-95`) and the GX runner's shape, **aggregate scalars/summaries, not row values**: `observed_value = sanitize_json(outcome.observed_value)` is populated from GX's observed-value dict (e.g. `{"observed_value": 5}` — a count/percentage), and for an errored check it's just `{"error": &lt;message&gt;}` — explicitly called out as safe because "GX exception messages are schema-level (no row data)" (`run_service.py:69-71`). `expected_value` mirrors the check's config (e.g. `{"column": "id"}`) — schema metadata, not data. So the facet-building code should read `observed_value`/`expected_value`/`metric_value`/`status`/check `name`/`column`/`expectation_type` freely, but **must never read `sample_failures`** — that field alone carries potential PII/row values and is exactly what the OL `DataQualityAssertionsDatasetFacet`/metrics facet must stay clear of, matching the acceptance criterion "no sample rows in events."

One paragraph for the spec: nothing that reaches an OpenLineage RunEvent may include `Result.sample_failures` (partial_unexpected_list / unexpected_index_list — real row values, the only PII-bearing result column) or any raw exception text (`run.failure_reason` is already a fixed, classified, redaction-safe string via `classify_failure_reason`, `app/services/failure_classifier.py` — safe to include; a raw caught exception's `str(exc)` is not, mirroring the "never surface raw adapter text" rule already enforced in `execute_run`'s except-block, `run_service.py:235-240`). Everything else the facets need — check name, column (from `check.config.get("column")`), expectation_type, status, metric_value, observed/expected aggregate JSON — is schema/aggregate-only and safe by the same reasoning `run_service.py`'s own docstrings already establish for those fields.

## 7. Test patterns for the worker/services layer

`run_service` is tested with **no DB, no GX**: a `FakeSession` (`tests/services/test_run_service.py:20-49`) records `add_all`'d rows and counts `commit`/`rollback`, plus a `refresh` stub that can simulate a concurrent status change; a `FakeRunner`/`FakeMonitorRunner` (lines 62-147ish) returns canned `SuiteOutcome`/`CheckOutcome` or raises. Tests call `run_service.execute_run(_sess(session), run=run, checks=checks, runner=runner, table=..., schema=...)` directly (e.g. `test_successful_run_persists_results_and_marks_succeeded`, lines 189-224; `test_runner_exception_marks_failed_and_persists_no_results`, lines 267-282) and assert on `run.status`, `session.added`, `session.commits`/`rollbacks`. **This is the harness an OL-emit hook inside `execute_run`/`skip_run` would be tested through** — a fake/spy "emitter" (or a monkeypatched `emit` function) injected the same way `FakeRunner` is, asserting it was called with the right `RunEvent` shape and that a raised exception from it doesn't propagate out of `execute_run`.

The **"no network by default" / fail-open assertion pattern** already exists for the alerting publishers, and is the direct template to reuse:



For the OL emitter, the equivalent would monkeypatch the lazily-imported `OpenLineageClient`/its `.emit` (or the module-level client factory) and assert it's never constructed/called when no transport env is set — plus a **fail-open** test modeled on `run_service.py`'s own `test_runner_exception_marks_failed_and_persists_no_results` (lines 267-282): inject an emitter whose call raises, and assert `execute_run`'s return value / `run.status` / `session.commits` are completely unaffected (mirroring `alert_dispatch.publish_run_outcome`'s own `except Exception: log.exception(...); return False` at `app/alerting/dispatch.py:59-61`, which is the canonical "best-effort, never raises, run already persisted" wrapper to copy for the OL emit call sites).

## 8. Celery worker lifecycle — client construction

There is **no per-worker-boot heavyweight-client-init hook currently in use** for this class of dependency — the repo's existing precedent (secret store, OTel exporters, result publisher) is a **process-wide lazy singleton built on first use and cached**, not a Celery signal-based init:

- `app/alerting/registry.py:29-77` — `get_result_publisher()` builds a `CompositePublisher` once (`_publisher_singleton`, guarded by `_publisher_lock`), with `reset_result_publisher_cache()` for tests.
- `app/core/secrets.py` — `AzureKeyVaultStore`/`RedisSecretStore` lazily build their client on first `.get`/`.set` call (`_client_lazy`, guarded by a per-instance `threading.Lock`), not at import or worker-boot time; `get_secret_store()` (referenced throughout `tasks.py`, e.g. line 125) is itself presumably an `lru_cache`d factory (same shape as `config.get_settings`, `app/core/config.py:203-205`, `@lru_cache(maxsize=1)`).

Celery **does** use signals for other cross-cutting concerns — `celery_app.py:16-23` imports `beat_init`, `before_task_publish`, `setup_logging`, `task_postrun`, `task_prerun`, `worker_process_init` from `celery.signals` (used elsewhere, e.g. `configure_logging`/`configure_tracing`/`instrument_celery` at worker boot, and the request-id correlation on `before_task_publish`/`task_prerun`/`task_postrun`) — so `worker_process_init` is an available, already-imported hook if the OL client needs one-time-per-worker-process setup (e.g. registering an `os.register_at_fork` handler the way the OTel `BatchLogRecordProcessor` does per `logging.py:114-118`'s comment on fork-safety). But **the norm for a config-gated, possibly-absent client in this codebase is a lazily-built, `lru_cache`/module-level-singleton-with-lock, not a Celery signal** — the spec should follow `get_secret_store`/`get_result_publisher`'s shape (a `get_openlineage_client() -&gt; OpenLineageClient | None` cached singleton) rather than introducing a new signal-based init path.</result>
